Contenedor de desarrollo

Se adoptó Docker Compose como plataforma de contenerización para garantizar un entorno homogéneo entre todos los miembros del equipo y evitar problemas de compatibilidad entre máquinas.

# Levantar el contenedor
docker compose -f \
  docker/docker-compose.jhipster.yml up -d

# Acceder al shell del contenedor
./jhipster-shell.sh

# Crear directorio de la aplicación
mkdir /evergreen/distribucion
cd    /evergreen/distribucion
Stack del contenedor
1
Eclipse Temurin JDK 21 — imagen Docker personalizada por incompatibilidad con imágenes oficiales (Java 17)
2
Node.js 22 — runtime para el generador y el frontend
3
JHipster CLI[1] — instalado globalmente vía npm
4
Volúmenes persistentes — cachés de Maven (.m2) y npm para acelerar re-generaciones
5
Git — control de versiones para rastrear cambios por historia de usuario
⚠ Hallazgo técnico
Las imágenes oficiales de Docker incluían Java 17, incompatible con JHipster 8.x que requiere Java 21. Se construyó una imagen personalizada basada en Eclipse Temurin JDK 21.

Entidad Pedido y sus relaciones
Atributos de Pedido
entity Pedido {
  identificador String    required unique
  fechaEntrada  LocalDate required
  fechaSalida   LocalDate required
  estado        String
}

// Relaciones
relationship ManyToOne {
  Pedido{cliente(nombre) required}
    to Cliente
}
relationship ManyToMany {
  Pedido{producto(nombre)} to
    Producto{pedido(identificador)}
}
relationship OneToMany {
  CanalComercializacion{pedido} to
    Pedido{canalComercializacion(nombre)}
  Transporte{pedido} to
    Pedido{transporte(matricula)}
}
ManyToOne → Cliente ManyToMany ↔ Producto OneToMany ← Canal OneToMany ← Transporte
Decisiones de diseño
Relación ManyToOne
Pedido → Cliente
Un pedido pertenece a un único cliente. Campo de visualización: nombre. Marcado como required.
Relación ManyToMany (bidireccional)
Pedido ↔ Producto
Genera tabla intermedia PEDIDO_PRODUCTO automáticamente. Navegable en ambas direcciones.
Relación OneToMany
Canal / Transporte → Pedido
Un canal o transporte puede estar asociado a múltiples pedidos. La FK queda en la tabla PEDIDO.
⚠ Restricción JDL[3]
Las relaciones bidireccionales requieren que todas las entidades involucradas estén en el mismo archivo JDL.

Lote separado, cantidad, ubicación en bodega y responsable asignado. También vinculada a un pedido mediante ManyToOne.

Sección 06 - Incompatibilidades NTFS - Docker Linux
Desafíos en entornos Windows
Problema 1 - Line endings
CRLF vs LF
Archivos generados en Windows con formato CRLF (\r\n) provocaban errores en scripts dentro del contenedor Linux (Maven Wrapper). Solución: normalizar a LF (\n) y configurar reglas en .gitattributes.
Problema 2 - Volúmenes Docker
Sobreescritura de .m2
Al montar volúmenes sobre el repositorio local de Maven (.m2), Docker sobreescribía la estructura interna, causando fallos en resolución de dependencias. Solución: montar únicamente subdirectorios específicos.
Problema 3 - Rendimiento node_modules
Degradación en archivos pequeños
La sobrecarga de traducción entre NTFS y el filesystem Linux del contenedor degrada operaciones sobre múltiples archivos pequeños. Solución: usar volúmenes nombrados gestionados por Docker (filesystem nativo Linux).
Problema 4 - Permisos
Directorios asignados a root
Directorios creados dentro de volúmenes asignados al usuario root, impidiendo modificaciones por el usuario de JHipster. Solución: reasignación explícita con chown o scripts de inicialización.
Problema 5 - Red Docker
IPs internas no accesibles desde host
Las IPs internas de contenedores Docker no son accesibles desde el host en Windows (a diferencia de Linux). Solución: exposición explícita de puertos en docker-compose.yml para acceso por localhost.

Sección 06 - Hallazgos del proceso de generación
Aprendizajes del ciclo JDL
1
JDL único y acumulativo
No es viable usar archivos JDL[3] separados por HU cuando existen relaciones bidireccionales. El archivo modelo-completo.jdl creció con cada historia evitando conflictos de regeneración.
2
DTOs en relaciones bidireccionales
Cuando existen relaciones bidireccionales, todas las entidades involucradas deben incluirse en las opciones de generación de DTOs para evitar errores de compilación en los mappers.
3
Limpiar H2 tras --force
Al regenerar con la bandera --force, es necesario limpiar la base de datos H2 (rm -rf target/h2db) para evitar conflictos de checksums en Liquibase.
4
Git por historia de usuario
El versionamiento con Git permitió rastrear con precisión los cambios introducidos en cada historia de usuario, facilitando la revisión y el rollback si fuera necesario.
5
Java 21 en imagen Docker
Las imágenes Docker oficiales de JHipster[1] incluían Java 17, incompatible con JHipster 8.x. Se construyó una imagen personalizada basada en Eclipse Temurin JDK 21.
6
Linux preferido sobre Windows
JHipster[1] es funcional en Windows pero su ejecución via Docker requiere configuración adicional de volúmenes, permisos y red. Linux ofrece un comportamiento más predecible y mejor rendimiento.

Sección 07 - Síntesis del caso de estudio
Conclusiones
→
Productividad en scaffolding
JHipster[1] generó en ~15 segundos una aplicación con 52.339 LOC en 348 archivos, cubriendo backend, frontend, base de datos, autenticación JWT, internacionalización y configuración Docker. El mayor aporte se concentró en las etapas iniciales del proyecto.
→
Desarrollo incremental efectivo
El enfoque de JDL[3] único y acumulativo permitió implementar las 3 historias de usuario de forma ordenada, sin perder entidades previas ni relaciones ya establecidas entre ellas.
→
Código con buenas prácticas
El código generado implementó patrones recomendados (DTOs, servicios con interfaces, mappers, Liquibase) y siguió una estructura estandarizada que facilitó su comprensión y extensión manual.
→
Curva de utilidad decreciente
La proporción de código generado vs. escrito manualmente disminuyó progresivamente al avanzar en la lógica de negocio. JHipster[1] es más valioso al inicio y menos determinante en fases de implementación específica.
